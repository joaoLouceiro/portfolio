Overview

Project Alpha started from a simple frustration: every collaborative editing tool I'd used either locked documents during edits, lost changes under load, or required a complex OT implementation that only three people in the world truly understood. I wanted something simpler — and more correct.

The result is a platform built around a conflict-free replicated data type (CRDT) engine written from scratch in Go. Any number of users can edit the same document simultaneously. Changes merge automatically, deterministically, and without a central coordinator making decisions about conflict resolution.

The problem

Traditional collaborative editors use one of two approaches: operational transformation (OT), which requires a central server to sequence operations, or last-write-wins (LWW), which silently discards data under concurrent edits. Neither was acceptable.

CRDTs give you strong eventual consistency without coordination. Every replica converges to the same state once all operations have propagated — no server needed to arbitrate.

The tradeoff with CRDTs is usually metadata overhead: each character carries a vector clock. For a document editor, this can grow unbounded. A core part of this project was designing a compact identifier scheme that keeps metadata size sub-linear relative to document length.

Architecture

The system is intentionally simple at the transport layer. Clients connect over WebSockets and receive a stream of CRDT operations. The server is a thin relay — it broadcasts operations and persists them to PostgreSQL for durability. It doesn't understand documents; it only understands operations.

Because the CRDT logic lives entirely in the client, the server can scale horizontally without coordinating state. New nodes join the cluster, subscribe to the Postgres WAL via logical replication, and immediately serve clients. No distributed locks, no leader election.

The CRDT implementation

The data structure is a variant of a Logoot sequence CRDT, extended with a tombstone-free garbage collection scheme. Characters are identified by a fractional position between their neighbours, represented as a base-32 string to keep comparisons fast. 

type Char struct {
    ID       Identifier // fractional position
    AuthorID string     // for GC attribution
    Value    rune       // the actual character
    Deleted  bool       // soft-delete flag
}

// Insert places a new char between left and right neighbours.
func (doc *Document) Insert(left, right Identifier, value rune) Char {
    id := doc.allocBetween(left, right)
    c  := Char{ID: id, AuthorID: doc.clientID, Value: value}
    doc.integrate(c)
    return c
}

Persistence & replay

Every operation is appended to an operations table in PostgreSQL. New clients joining a document replay the full operation log to reconstruct state. For large documents, a periodic snapshot compresses the log — clients load the snapshot and only replay operations that arrived after it.

Snapshot creation runs as a background job triggered when the operation log for a document exceeds 10,000 entries. Snapshots are stored as JSONB columns alongside a watermark operation ID for efficient replay.

Lessons learned

Building this surfaced a few things that surprised me even after reading extensively about CRDTs beforehand.

  • Tombstones are unavoidable at scale. My first design avoided them; within a week of real usage the GC scheme was causing subtle reordering bugs under high concurrency. Tombstones came back.
  • The WebSocket reconnect path is the hardest part. Handling partial sync on reconnect — figuring out exactly which operations a client missed — required more careful design than the CRDT itself.
  • Postgres is remarkably good at this workload. Append-only operation logs with a covering index on (document_id, sequence_number) handle tens of thousands of writes per second on modest hardware.

What's next

The next milestone is rich text support — currently the editor handles plain text only. Extending the CRDT to support formatting spans (bold, italic, links) without breaking the existing merge semantics is the current design challenge. I'm exploring a peritext-inspired approach that annotates ranges rather than individual characters.

The source is open on GitHub. Issues, pull requests, and especially challenging test cases are welcome.
