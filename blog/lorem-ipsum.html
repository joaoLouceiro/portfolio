Why I stopped reaching for React

It started with a greenfield project and a stubborn question: what if I didn't? Not as an exercise in minimalism, but as a genuine architectural choice. Three months later, here's what I found.

Opinion Architecture Hypermedia Datastar

Every new project starts the same way. You open a terminal, you type npm create vite@latest, and before you've written a single line of business logic you already have a framework, a bundler, a linting config, and enough node_modules to fill a hard drive. I did this for years without questioning it.

Then came a project where I didn't have the luxury of that ritual. Tight deadline, limited hosting, a backend colleague who hadn't written JavaScript since 2012. The question surfaced almost by accident: does this actually need React?

The experiment

I decided to build the whole thing with Datastar — a hypermedia framework that leans into server-sent events and HTML-first thinking. No virtual DOM. No component lifecycle. No hydration mismatch errors at 11 PM on a Friday.

The stack was deliberately boring: a Go server, PostgreSQL, plain HTML templates, and Datastar for the interactive bits. Here's what the main entry point looked like: 

func main() {
    mux := http.NewServeMux()
    mux.HandleFunc("GET /", handleIndex)
    mux.HandleFunc("GET /api/posts", handlePosts)
    mux.HandleFunc("POST /api/contact", handleContact)

    log.Println("Listening on :8080")
    http.ListenAndServe(":8080", mux)
}

That's it. No webpack. No tree-shaking. No SSR configuration. Just a server that serves HTML and responds to fetch requests with more HTML — or in Datastar's case, server-sent events that patch the DOM.

What surprised me

Three things caught me off guard, in roughly this order of surprise.

1. It was genuinely fast

Not "fast enough" — actually fast. No client-side JavaScript bundle to parse and execute before the user sees anything. The first meaningful paint happened in under 200ms on a 3G throttled connection. React's reconciler never ran, because there was no reconciler.

The fastest JavaScript is the JavaScript you don't ship. The second fastest is the JavaScript you ship but don't execute until needed. Hypermedia is mostly the first one.

2. Forms just worked

This one felt almost embarrassing. I've spent hours debugging controlled component state, form validation timing, and onChange vs onBlur semantics in React. With Datastar and standard HTML form elements, the browser just handled it. Validation attributes, tab order, accessibility — all there by default. 

<form data-on-submit="@post('/api/contact')">
  <input
    type="email"
    name="email"
    required
    placeholder="you@example.com"
  />
  <textarea name="message" minlength="20" required></textarea>
  <button type="submit">Send</button>
</form>

The server responds with an SSE fragment that replaces the form with a success message. Progressive enhancement for free — if JavaScript is disabled, the form still submits normally.

3. Complexity didn't accumulate

This is the one I think about most. In React projects, complexity has a way of compounding. State management starts simple, then you need global state, then you need derived state, then you need memoisation to stop re-renders. Each solution introduces new abstractions.

The hypermedia approach kept complexity local. Each page was dumb by default. Interactivity lived at the edges, where it was genuinely needed. The mental model never expanded beyond "click button, server does thing, page updates."

Worth noting: Datastar is not HTMX. It uses signals and a reactive store model alongside its hypermedia primitives. You get fine-grained reactivity where you want it, server-driven updates where you don't.

Where React still wins

I'm not here to bury React. There are real cases where a rich client-side component model is the right tool:

  • Complex offline-capable applications with heavy local state
  • Data visualisations with high-frequency DOM updates
  • Collaborative real-time editors where latency to the server matters
  • Large teams where component encapsulation aids code organisation

But notice how specific those cases are. The majority of web applications — CRUD interfaces, content sites, dashboards, contact forms, e-commerce flows — don't live in that list.

The conclusion

Three months on, the project is in production. It's fast, it's maintainable, and my backend colleague understands every line of it. We've shipped three new features without touching a package.json.

The lesson isn't "React bad." It's that every tool carries assumptions, and the most dangerous assumptions are the ones you stopped questioning. Question them. Sometimes you'll reach for React anyway — and that's fine. But at least it'll be a choice.
